# 数据分析控制台下钻与明细查看功能改造方案

## 背景与现状

当前系统的 BI 控制台在数据下钻与明细查看逻辑上存在前后端不一致、架构不合理的问题：
1. **前端正则解析脆弱**：前端使用正则表达式强行解析 Raw SQL（提取 SELECT、FROM）来拼接查看明细的查询语句。遇到子查询、JOIN、CTE 等复杂 SQL 时会直接崩溃。
2. **SQL 注入风险**：前端直接将表格行数据拼接成 `WHERE` 条件发送回后端，极易引发安全问题。
3. **后端 AST 闲置**：后端精心设计的 `DrillDownRequest` 和 AST 解析逻辑完全没有被前端调用。前端的“下钻”实际上是“查看明细（Drill-through）”，而后端的“下钻”是真正的“多维聚合下钻（Drill-down）”。

## 改造目标

1. **废除前端 SQL 解析**：前端不再处理任何字符串级别的 SQL 拼接与解析。
2. **前后端职责分离**：后端承担所有 SQL 解析、安全构建的任务，暴露标准的 API 供前端调用。
3. **引入专业解析器**：后端引入 `sqlglot` 将原始 SQL 安全、准确地解析为上下文（Context），并以此为基础进行后续的明细查询。
4. **安全加固**：使用参数化或安全的过滤构建方式，杜绝 SQL 注入。

## 详细设计与实施步骤

### 阶段一：后端 API 改造与 SQL 解析引入

1. **依赖引入**：
   - 在后端引入 `sqlglot` 库，用于解析前端传来的 Raw SQL。
2. **改造现有 `/query/raw` 接口**：
   - 接口职责保持不变（执行前端输入的 SQL）。
   - 返回结果需增强：除了 `columns` 和 `data` 外，后端利用 `sqlglot` 提取 SQL 中的基础表名（`base_table`）以及原始条件，作为一个安全的 `context` 返回给前端。
3. **新增 `/drill-through` (查看明细) 接口**：
   - **入参定义 (`DrillThroughRequest`)**：
     - `raw_sql`: 用户原始执行的 SQL 语句。
     - `filters`: 前端传递的当前行的维度数据字典（`Dict[str, Any]`），作为额外的过滤条件。
     - `clicked_metric`: (新增) 用户当前点击的指标列别名，用于智能推断查询主体。
     - `limit` & `offset`: 分页参数。
   - **执行逻辑**：
     - 后端接收到 `raw_sql` 后，通过 `sqlglot` 解析出基础的 AST（提取 FROM, JOIN 和原有的 WHERE 条件）。
       - **智能投影 (Smart Projection)**：根据 `clicked_metric` 的表达式类型动态决定查询内容。若为 `COUNT(DISTINCT table.field)`，则明细语句的投影变为 `SELECT DISTINCT table.*`，返回该去重主体的完整上下文属性；若是普通聚合，则退化为 `SELECT *` 获取底层大宽表。
     - 将前端传来的 `filters` 安全地转化为 AST 上的过滤条件，并与原有的 WHERE 条件进行 AND 合并。
     - 剥离原 SQL 中的 `GROUP BY`, `ORDER BY`, `LIMIT`。
     - 生成总数查询和明细查询的两条新 SQL 在数据库中执行，返回给前端。

### 阶段二：数据源适配器 (Adapter) 安全加固

1. **安全构建**：
   - 在 `ClickHouseAdapter` 中，涉及 AST 转 SQL 或拼接的地方，需要确保对字段名和值的有效转义。由于 ClickHouse 驱动的支持度限制，在组装 `drill-through` 的查询时，利用 `sqlglot` 原生提供的构建能力来生成安全的方言 SQL。

### 阶段三：前端重构

1. **移除正则逻辑**：
   - 删除 `BiDashboard.vue` 中所有关于 `splitSelectClause` 以及针对 SQL 字符串匹配的逻辑。
2. **识别指标列的新逻辑**：
   - （可选方案）指标列识别可以交由后端在 `/query/raw` 时一并解析返回（例如后端解析出哪些别名对应了聚合函数）；或者前端在结果集中通过简单的数据类型与表头名称做启发式判断（暂时保持前端基于列名包含 count/sum 等标识的简单判断作为平滑过渡）。
3. **调用新接口**：
   - 当点击可下钻的单元格时，收集同行其他维度的 Key-Value。
   - 带着 `raw_sql` 和这批 `filters`，直接请求后端的 `/drill-through` 接口。
   - 弹窗的分页逻辑直接对接 `/drill-through` 的 `limit/offset` 入参。

## 预期效果

- 用户可以安全、稳定地在复杂的（如包含 JOIN）查询结果上点击指标查看明细。
- 彻底消灭前端浏览器的 `SyntaxError` 和后端的 SQL 注入隐患。
- 架构回归健康，前端只做视图呈现和交互状态管理，数据拼装统一收敛在后端。
