# 同源SQL支持多版本表切换设计草案 (Table Mapping Design)

## 背景需求
在真实的数仓环境中，由于每日快照、分表策略等原因，底层物理表经常会存在多个版本（如 `order_3_1`, `order_3_2` 等）。
为了让固化的 SQL 组件（Widget）与数据看板（Dashboard）具备更强的生命力，必须实现**同一套 SQL 动态应用到不同物理表版本**上的能力，避免每天重建看板或重写长篇的 JOIN SQL。

## 最终落地架构方案 (Implemented Pre-Compilation Macro System)

我们在系统中实现了一套基于 **服务层安全预编译 (Secure Pre-Compilation)** 的宏替换架构，彻底解决了不同 SQL 方言解析器由于 `{}` 花括号产生的语法冲突。

### 1. 语法冲突与弃用 AST 替换的原因
最初计划使用 `sqlglot` 的 AST 在底层进行 `Table` 节点的拦截替换。但在实际应用中发现：
- 在 ClickHouse 等方言中，紧贴在一起的大括号 `{}` 会被解析器错误地判定为 `Map/Dictionary` 字面量。
- 这会导致 `FROM order_{{version}}` 无法被正确识别为一个表（被解析为 `VarMap` 节点），从而引发后续的所有 AST 分析崩溃。

### 2. 重构为安全的前置宏引擎
为了规避该问题并大幅拓宽宏的使用场景（不仅支持换表，还支持动态过滤），我们将宏处理逻辑提升到了更上层的 `QueryService` 中：

1. **预处理安全校验**：在把包含 `{{var}}` 的 SQL 丢进数据库或底层解析器之前，由服务端对其执行字符串层面的编译替换。
2. **防注入层 (Sanitization)**：为了防止简单的字符串替换带来 SQL 注入灾难，系统对前端传来的宏变量 `value` 实施了极其严格的正则白名单约束（仅允许：英文字母、数字、下划线、短横线、点号以及中文）。若检测到包含单引号、分号等可能破坏 SQL 结构的危险字符，直接在业务层熔断报错。
3. **透传至安全 AST**：预编译完成后，生成了一句干干净净的标准 SQL (如 `SELECT * FROM order_3_1 WHERE country = '中国'`)。此时再交给底层的 `ClickHouseAdapter` 进行 `sqlglot` 语法树解构与下钻计算，各司其职，完美衔接。

### 3. 应用场景演示
- **动态表版本映射**：`SELECT * FROM bi_demo.order{{version}}` 结合宏参数 `version: s`，无缝查询真实的 `orders` 表。
- **全局参数级联**：大屏顶部配置一个参数 `cty: 中国`，看板下方的所有图表若带有 `WHERE country = '{{cty}}'` 均会自动高亮刷新，实现典型的全局动态筛选能力。
